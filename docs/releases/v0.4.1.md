# v0.4.1 — JWT-pair credentials for /fetch and /hydrate-threads

`bsky-saves serve` now accepts JWT-pair credentials on `/fetch` and `/hydrate-threads`, alongside the existing app-password shape. This unblocks the `bsky-saves-gui` integration: the GUI's session state is a stored JWT pair (the app password isn't kept after first sign-in), and v0.4.0's app-password-only requirement was a structural blocker.

## New credential shape

    {
      "credentials": {
        "access_jwt": "<jwt>",
        "refresh_jwt": "<jwt>",
        "did": "did:plc:...",
        "pds": "https://bsky.social"
      }
    }

The daemon detects which shape was sent (`app_password` present → app-password path; `access_jwt` present → JWT path; neither → 400 missing credentials). `did` is treated as an opaque string — no JWT decoding, no `sub`-claim verification. `pds` is optional and defaults to `https://bsky.social`.

## /fetch refresh handling (JWT path)

On upstream 401 (typically expired `access_jwt`), the daemon calls `com.atproto.server.refreshSession`, retries once with the new tokens, and returns the rotated pair to the GUI in the response's new `rotated_credentials` field:

    {
      "saves": [...],
      "cursor": "...",
      "rotated_credentials": {
        "access_jwt": "<new>",
        "refresh_jwt": "<new>",
        "did": "did:plc:..."
      }
    }

The GUI **must** persist `rotated_credentials` over its stored JWT pair when present — AT Protocol invalidates a `refresh_jwt` once it's been used to mint a new pair. Failing to persist the rotation leaves the stored `refresh_jwt` silently dead, and the next refresh will fail with `auth refresh failed`.

`rotated_credentials` is **always absent** on:
- App-password path responses.
- JWT-path responses where no refresh was needed.
- All `/hydrate-threads` responses (no upstream call could trigger refresh).

### Refresh failure modes

    // refreshSession itself failed (refresh_jwt invalid/expired/revoked):
    401 { "error": "auth refresh failed", "code": "refresh_failed" }

    // refreshSession succeeded but the retried upstream call still got 401:
    401 { "error": "auth refresh failed", "code": "upstream_rejected_after_refresh" }

The `error` string is the GUI-facing contract and is identical for both modes (the GUI handles both the same way: re-prompt for app password). `code` is informational/diagnostic.

### Non-401 failures stay on the existing path

Non-401 upstream failures (500, transport errors) take the existing silent-fallback re-probe path. No refresh is attempted under either credential variant — refresh wouldn't help and would burn a refresh token.

## /hydrate-threads JWT path

Skips credential validation entirely. The endpoint's upstream call is to the public AppView unauthenticated; the JWT is unused. No `createSession` round-trip, no `getSession` validation. The JWT is accepted but not checked — a typo'd token still produces correct-because-unauthenticated thread reads. Security on this endpoint comes from the origin allowlist plus 127.0.0.1 binding (same as `/enrich`).

## /ping capability advertisement

The `/ping` `features` array gains a new entry, `"jwt-credentials"`:

    {
      "name": "bsky-saves",
      "version": "0.4.1",
      "features": [
        "fetch-image", "extract-article", "fetch", "enrich", "hydrate-threads", "jwt-credentials"
      ]
    }

This lets consumer routing layers use uniform feature-flag introspection across all capabilities, instead of mixing feature-array detection (for endpoints) with version-string parsing (for the JWT-pair path).

## Backwards compatibility

Purely additive. **No breaking changes.**

- App-password credential shape continues to work unchanged on both endpoints. Existing v0.4.0 callers don't need to update.
- `/enrich`, `/ping`, `/fetch-image`, and `/extract-article` are unmodified (other than the new `features` array entry on `/ping`).
- The new `rotated_credentials` field on `/fetch` responses is optional and only present when a refresh occurred during the request — old clients that don't read it are unaffected.

## Internal changes

- `bsky_saves.fetch._DirectEndpointFailedError` and `NoBookmarkEndpointError` now carry structured status-code attributes (`status_code: int | None` and `status_codes: list[int]` respectively) so the daemon's `/fetch` handler can discriminate 401-from-upstream (refresh + retry) from non-401 failures (silent fallback re-probe).
- New `bsky_saves.auth.refresh_session(pds_base, refresh_jwt)` helper — thin wrapper over `com.atproto.server.refreshSession`. Raises `httpx.HTTPStatusError` on non-2xx (e.g. `400 ExpiredToken` when the refresh_jwt itself is dead).
- `bsky_saves.serve._validate_creds` now returns a normalised dict with a `variant` discriminator (`"app_password"` or `"jwt"`); other internal helpers branch on this.
